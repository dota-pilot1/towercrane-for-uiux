# 메일 서버 설정에 대해

이 문서는 Towercrane Prototype Console의 이메일 인증/비밀번호 찾기 메일 설정 위치와 BeautyBook과의 차이를 정리한다. 실제 메일 계정, 앱 비밀번호, AWS 키 같은 Secret 값은 문서에 기록하지 않는다.

## 결론

현재 운영에서 보인 `Internal server error`는 프론트 `.env.production` 문제가 아니라 백엔드 운영 서버의 Gmail SMTP 인증 문제다.

운영 로그 기준 Gmail이 Towercrane 서버의 SMTP 로그인 값을 거부했다.

```text
535-5.7.8 Username and Password not accepted
code=EAUTH
command=AUTH LOGIN
```

즉 서버가 Gmail까지는 접속했지만, `MAIL_USERNAME`/`MAIL_PASSWORD` 조합이 Gmail에서 유효하지 않다. 보통 원인은 다음 중 하나다.

- Gmail 앱 비밀번호가 틀렸거나 만료됨
- 일반 Gmail 비밀번호를 넣음
- 해당 Google 계정의 2단계 인증/앱 비밀번호 설정이 변경됨
- 운영 서버 `.env`에 BeautyBook에서 쓰는 정상 메일 계정과 다른 값이 들어감

## `.env.production`과의 관계

프론트 `.env.production`은 브라우저 번들에 들어가는 공개 설정이다. API 주소 같은 값만 둔다.

BeautyBook 프론트:

```env
NEXT_PUBLIC_API_BASE_URL=https://dxline-tallent.com
```

Towercrane 프론트:

```env
VITE_API_BASE_URL=https://api.hibot-docu.com/api
```

메일 서버 설정은 여기에 두지 않는다. Gmail 계정과 앱 비밀번호는 Secret이므로 백엔드 운영 서버의 `.env`에만 있어야 한다.

## Towercrane 운영 메일 설정 위치

EC2:

```bash
ssh -i docs-for-배포/hibot-d-server-key.pem ubuntu@54.180.215.129
cd ~/towercrane/towercrane-for-uiux-server
vi .env
```

필수 값:

```env
MAIL_DEV_MODE=false
MAIL_HOST=smtp.gmail.com
MAIL_PORT=587
MAIL_SECURE=false
MAIL_USERNAME=<gmail_address>
MAIL_PASSWORD=<gmail_app_password>
MAIL_FROM_NAME=Towercrane Prototype Console
MAIL_FROM_ADDRESS=<gmail_address>
```

`MAIL_PASSWORD`는 Google 계정 비밀번호가 아니라 Gmail 앱 비밀번호여야 한다.

`MAIL_AUTH_METHOD`는 선택값이다. 지정하지 않으면 Nodemailer가 서버 응답에 맞춰 인증 방식을 고른다. 특별한 이유가 없으면 비워두는 편이 BeautyBook의 JavaMail 설정과 더 비슷하다.

## BeautyBook과 비교

BeautyBook은 Spring Boot의 JavaMailSender를 사용한다.

설정 위치:

- `/Users/terecal/beauty-book-hair/beauty-book-server/src/main/resources/application.yaml`
- `spring.mail` 블록

```yaml
spring:
  mail:
    host: smtp.gmail.com
    port: 587
    username: ${MAIL_USERNAME:}
    password: ${MAIL_PASSWORD:}
    properties:
      mail.smtp.auth: true
      mail.smtp.starttls.enable: true
```

여기서 `username: ${MAIL_USERNAME:}`와 `password: ${MAIL_PASSWORD:}`는 환경변수 주입이다. 기본값은 빈 문자열이므로 로컬에서 환경변수를 안 넣고 기동하면 메일 발송 시 인증에 실패할 수 있다.

BeautyBook 로컬에서 메일 발송을 테스트하려면 셋 중 하나로 주입한다.

- 셸에서 `export MAIL_USERNAME=...`, `export MAIL_PASSWORD=...` 후 실행
- IDE Run Configuration의 환경변수로 추가
- `application-local.yaml`에 직접 추가

`application-local.yaml`에 Secret을 적었다면 커밋하지 않는다.

BeautyBook 운영 서버는 EC2의 `/home/ubuntu/.env`에서 `MAIL_USERNAME`/`MAIL_PASSWORD`를 주입받는다. 로컬의 `배포 가이드/.env.prod`에는 메일 값이 없고, 프론트 `.env.production`도 메일과 무관하다.

운영 위치:

```text
EC2: ubuntu@13.209.195.64
File: /home/ubuntu/.env
```

운영 키:

```env
MAIL_USERNAME=<실제 Gmail 주소>
MAIL_PASSWORD=<Gmail 앱 비밀번호>
```

BeautyBook 배포 스크립트는 백엔드 기동 전에 대략 다음 방식으로 이 값을 로드한다.

```bash
set -a && source ~/.env && set +a
```

실제 값 확인 시에는 앱 비밀번호가 노출되지 않게 마스킹해서 본다.

```bash
ssh -i "/Users/terecal/beauty-book-hair/배포 가이드/hibot-d-server-key.pem" ubuntu@13.209.195.64 \
  "grep -E '^(MAIL_USERNAME|MAIL_PASSWORD)=' ~/.env | sed -E 's/^(MAIL_USERNAME=).*/\1<redacted>/; s/^(MAIL_PASSWORD=).*/\1<redacted>/'"
```

Towercrane도 구조는 같다.

| 항목 | BeautyBook | Towercrane |
|---|---|---|
| SMTP 서버 | `smtp.gmail.com:587` | `smtp.gmail.com:587` |
| TLS 방식 | STARTTLS | STARTTLS |
| 계정 주입 | EC2 서버 `.env` | EC2 서버 `.env` |
| 프론트 `.env.production` | API 주소 | API 주소 |
| 문제 지점 | 정상 계정/앱비밀번호 사용 중 | Gmail이 운영 SMTP 인증값 거부 |

## 수정 절차

1. BeautyBook 운영 서버에서 실제 동작 중인 메일 계정이 무엇인지 확인한다. Secret 값은 화면 공유나 문서에 노출하지 않는다.

```bash
ssh -i "/Users/terecal/beauty-book-hair/배포 가이드/hibot-d-server-key.pem" ubuntu@13.209.195.64
grep -E '^(MAIL_USERNAME|MAIL_HOST|MAIL_PORT)=' ~/.env \
  | sed -E 's/^(MAIL_USERNAME=).*/\1<redacted>/'
```

2. Towercrane 운영 서버의 `.env`를 수정한다.

```bash
ssh -i docs-for-배포/hibot-d-server-key.pem ubuntu@54.180.215.129
cd ~/towercrane/towercrane-for-uiux-server
cp .env ".env.backup.$(date +%Y%m%d-%H%M%S)"
vi .env
```

3. 다음 값을 정상 Gmail 앱 비밀번호 기준으로 맞춘다.

```env
MAIL_DEV_MODE=false
MAIL_USERNAME=<정상 Gmail 주소>
MAIL_PASSWORD=<정상 Gmail 앱 비밀번호>
MAIL_FROM_ADDRESS=<정상 Gmail 주소>
MAIL_AUTH_METHOD=
```

4. 백엔드를 재시작한다.

Towercrane의 `.env`에는 `MAIL_FROM_NAME=Towercrane Prototype Console`처럼 공백이 있는 값이 있으므로 `set -a && source .env`를 쓰면 shell 파싱 오류가 날 수 있다. 배포 스크립트는 `.env`를 직접 파싱해서 PM2에 주입한다.

수동으로 메일 Secret만 반영할 때는 필요한 값만 읽어서 PM2 환경을 갱신한다.

```bash
getv(){ sed -n "s/^$1=//p" .env | tail -n1; }
MAIL_DEV_MODE="$(getv MAIL_DEV_MODE)" \
MAIL_HOST="$(getv MAIL_HOST)" \
MAIL_PORT="$(getv MAIL_PORT)" \
MAIL_SECURE="$(getv MAIL_SECURE)" \
MAIL_USERNAME="$(getv MAIL_USERNAME)" \
MAIL_PASSWORD="$(getv MAIL_PASSWORD)" \
MAIL_FROM_NAME="$(getv MAIL_FROM_NAME)" \
MAIL_FROM_ADDRESS="$(getv MAIL_FROM_ADDRESS)" \
MAIL_AUTH_METHOD="$(getv MAIL_AUTH_METHOD)" \
  pm2 restart towercrane-back --update-env
pm2 save
```

5. 발송 API를 확인한다.

```bash
curl -i -X POST "https://api.hibot-docu.com/api/auth/email/send-code" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com"}'
```

정상이라면 첫 요청은 성공하고, 바로 다시 누르면 5초 재발송 제한 메시지가 나와야 한다.

## 운영 중 확인 명령

메일 관련 환경변수 형태만 확인한다. Secret 값은 출력하지 않는다.

```bash
grep -E '^(MAIL_DEV_MODE|MAIL_HOST|MAIL_PORT|MAIL_SECURE|MAIL_AUTH_METHOD|MAIL_USERNAME|MAIL_FROM_ADDRESS)=' .env \
  | sed -E 's/^(MAIL_USERNAME=).*/\1<redacted>/; s/^(MAIL_FROM_ADDRESS=).*/\1<redacted>/'
```

PM2 로그에서 Gmail 인증 오류를 확인한다.

```bash
pm2 logs towercrane-back --lines 100
```

`.env`와 PM2 프로세스 환경이 같은 Secret을 보고 있는지도 확인한다. 값은 출력하지 않고 길이와 해시만 비교한다.

```bash
v=$(sed -n "s/^MAIL_PASSWORD=//p" .env | tail -n1)
p=$(pm2 env 0 | sed -n "s/^MAIL_PASSWORD: //p" | tail -n1)
printf ".env len=%s sha=%s\n" "${#v}" "$(printf "%s" "$v" | sha256sum | cut -d" " -f1)"
printf "pm2  len=%s sha=%s\n" "${#p}" "$(printf "%s" "$p" | sha256sum | cut -d" " -f1)"
```

두 해시가 다르면 `.env`를 수정했더라도 PM2가 예전 환경변수를 들고 있는 상태다. 이 경우 위의 PM2 재시작 명령으로 다시 주입한다.

대표 오류:

| 로그 | 의미 |
|---|---|
| `535-5.7.8 Username and Password not accepted` | Gmail 계정/앱비밀번호 불일치 |
| `EAUTH` | SMTP 인증 실패 |
| `ETIMEDOUT` | SMTP 서버 접속 타임아웃 또는 네트워크 문제 |
| `Mail credentials are not configured` | `.env`에 `MAIL_USERNAME` 또는 `MAIL_PASSWORD` 누락 |

## 주의

- `MAIL_PASSWORD`는 절대 Git, 문서, 프론트 `.env.production`에 넣지 않는다.
- 운영 Secret은 EC2 서버 `.env` 또는 별도 Secret Manager에서 관리한다.
- Gmail 계정 비밀번호가 아니라 앱 비밀번호를 사용한다.
- 앱 비밀번호 변경 후에는 `.env` 수정만으로 끝나지 않는다. PM2 프로세스 환경까지 갱신해야 한다.
- 배포 시에는 `docs-for-배포`의 배포 가이드를 먼저 확인한다.
