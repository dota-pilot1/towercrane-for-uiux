# Towercrane 운영 배포 런북

이 폴더는 Towercrane 운영 배포에 필요한 문서만 모아둔 곳입니다. 일반적인 코드 업데이트는 아래 판단표로 대상 문서를 고른 뒤 진행합니다.

## 운영 구성

| 영역 | 운영 방식 | 문서 |
|---|---|---|
| 프론트엔드 | Vite 정적 빌드 → S3 업로드 → CloudFront 무효화 | `프론트엔드_배포_방법.md` |
| 백엔드 | EC2 `ubuntu` 서버 → NestJS 빌드 → PM2 재시작 | `백엔드_배포_방법.md` |
| 접속/환경변수 | 서버, 도메인, 버킷, 런타임 env 확인 위치 | `접속_및_환경변수_정보.md` |

## 어떤 것을 배포해야 하나

| 변경 내용 | 필요한 배포 |
|---|---|
| React 화면, CSS, 클라이언트 라우팅, API 호출 방식만 변경 | 프론트엔드만 |
| NestJS controller/service/schema, API 응답, DB 초기화/마이그레이션 변경 | 백엔드만 또는 백엔드 먼저 |
| 새 API를 프론트에서 호출 | 백엔드 배포 후 프론트엔드 배포 |
| `.env.production`의 `VITE_API_BASE_URL` 변경 | 프론트엔드 재빌드/재배포 |
| 서버 `.env`, SQLite 파일 경로, S3 업로드 env 변경 | 백엔드 재시작 |

## 배포 전 공통 체크

```bash
# 현재 변경 범위 확인
git status --short

# 프론트 타입 체크
cd towercrane-for-uiux-front
npm run typecheck

# 백엔드 빌드 체크가 필요한 경우
cd ../towercrane-for-uiux-server
npm run build
```

운영 서버는 GitHub의 `main`을 pull하는 방식으로 재배포합니다. 백엔드 배포가 필요한 변경은 먼저 원격 저장소에 반영되어 있어야 합니다.

## 표준 순서

1. 변경 범위 확인
2. 필요한 배포 대상 결정
3. 프론트엔드는 로컬에서 빌드 후 S3/CloudFront 배포
4. 백엔드는 EC2에서 `git pull`, `npm install`, `npm run build`, `pm2 restart`
5. 운영 URL/API 헬스 체크

## 권장 배포 명령

운영 배포는 수동 명령을 직접 조합하지 말고 repo의 스크립트를 우선 사용합니다.

```bash
# 전체 배포: 백엔드 → 프론트엔드
WAIT_FOR_INVALIDATION=true ./scripts/deploy-all.sh

# 프론트만 배포
WAIT_FOR_INVALIDATION=true ./scripts/deploy-frontend.sh

# 백엔드만 배포
./scripts/deploy-backend.sh
```

`deploy-all.sh`는 기본적으로 아래 조건을 강제합니다.

- 현재 브랜치가 `main`
- 로컬 워킹트리가 clean
- 로컬 `HEAD`가 `origin/main`과 동일

이유: 백엔드는 EC2에서 `origin/main`을 pull하고, 프론트는 로컬에서 빌드합니다. 둘이 다른 커밋을 기준으로 배포되면 화면과 API 계약이 어긋날 수 있습니다.

긴급 상황에서만 아래 옵션으로 검사를 우회합니다.

```bash
REQUIRE_CLEAN_GIT=false REQUIRE_SYNCED_MAIN=false ./scripts/deploy-all.sh
```

우회 배포는 운영에 반영된 커밋을 추적하기 어려워지므로 일반 배포에서는 사용하지 않습니다.

## 현재 스크립트 안정성 기준

- 백엔드 배포는 EC2에서 lock file을 잡아 동시 배포를 막습니다.
- 백엔드 배포 전 SQLite DB를 `data/backups/`에 백업합니다.
- 백엔드 배포 직전 커밋을 `/tmp/towercrane-backend-rollback-commit`에 기록합니다.
- 배포 후 헬스체크가 계속 실패하면 이전 커밋으로 checkout/build/restart 롤백을 시도합니다.
- 프론트는 hashed asset을 먼저 업로드하고 `index.html`을 마지막에 업로드합니다.
- `index.html`은 no-cache, asset은 immutable cache-control로 업로드합니다.
- 오래된 asset 삭제는 기본 비활성화입니다. 필요할 때만 `PRUNE_STALE_ASSETS=true`를 사용합니다.

## 보안 주의

- AWS Access Key, Secret Key, API Key를 문서에 직접 기록하지 않습니다.
- 로컬 AWS 인증은 `~/.aws/credentials` 또는 `aws configure`로 관리합니다.
- EC2 `.env`는 서버 내부에서만 관리합니다.
- `hibot-d-server-key.pem`은 권한을 `400`으로 두고 외부 공유하지 않습니다.
