# 프론트엔드 배포 방법

> 대상: `towercrane-for-uiux-front`  
> 방식: Vite production build → S3 sync → CloudFront invalidation

## 배포 정보

| 항목 | 값 |
|---|---|
| 운영 URL | `https://hibot-docu.com` |
| API Base URL | `https://api.hibot-docu.com/api` |
| S3 버킷 | `hibot-docu-front-hyun0316` |
| CloudFront Distribution ID | `E1L7QZR4LQV4LE` |
| CloudFront 도메인 | `d359limanz0pmj.cloudfront.net` |
| AWS 리전 | `ap-northeast-2` |
| AWS 인증 | 로컬 `~/.aws/credentials`의 `default` 프로파일 |

## 언제 프론트만 배포하나

- React 컴포넌트, 스타일, 클라이언트 상태, 라우팅만 바뀐 경우
- 이미 운영에 존재하는 API를 호출하는 방식만 바뀐 경우
- `.env.production`의 `VITE_API_BASE_URL`을 바꾼 경우

새 API나 백엔드 schema/service 변경이 필요한 기능이면 백엔드 배포를 먼저 진행합니다.

## 사전 확인

```bash
cd towercrane-for-uiux-front

# 운영 API URL 확인
cat .env.production

# AWS 인증 확인
aws configure list
aws sts get-caller-identity

# 가능하면 타입 체크
npm run typecheck
```

`.env.production`은 아래 값이어야 합니다.

```env
VITE_API_BASE_URL=https://api.hibot-docu.com/api
```

## 배포 명령어

권장 명령:

```bash
WAIT_FOR_INVALIDATION=true ./scripts/deploy-frontend.sh
```

스크립트는 다음 순서로 업로드합니다.

1. `dist/` 빌드
2. `index.html`을 제외한 hashed asset 먼저 업로드
3. `index.html`을 마지막에 no-cache로 업로드
4. CloudFront invalidation 생성
5. `WAIT_FOR_INVALIDATION=true`면 무효화 완료까지 대기

이 순서를 지키는 이유는 새 `index.html`이 먼저 배포되어 아직 올라가지 않은 asset을 참조하는 상황을 줄이기 위해서입니다.

수동 배포가 필요할 때만 아래 명령을 사용합니다.

```bash
cd towercrane-for-uiux-front

# 1. Vite 빌드
npx vite build

# 2. index.html 제외 asset 먼저 업로드
AWS_REGION=ap-northeast-2 \
aws s3 sync dist/ s3://hibot-docu-front-hyun0316 \
  --exclude "index.html" \
  --cache-control "public,max-age=31536000,immutable"

# 3. index.html 마지막 업로드
AWS_REGION=ap-northeast-2 \
aws s3 cp dist/index.html s3://hibot-docu-front-hyun0316/index.html \
  --cache-control "no-cache,no-store,must-revalidate" \
  --content-type "text/html"

# 4. CloudFront 캐시 무효화
AWS_REGION=ap-northeast-2 \
aws cloudfront create-invalidation \
  --distribution-id E1L7QZR4LQV4LE \
  --paths "/*"
```

`create-invalidation` 결과의 `Invalidation.Id`를 기록해 둡니다.

## 무효화 완료 확인

```bash
AWS_REGION=ap-northeast-2 \
aws cloudfront wait invalidation-completed \
  --distribution-id E1L7QZR4LQV4LE \
  --id <InvalidationId>

AWS_REGION=ap-northeast-2 \
aws cloudfront get-invalidation \
  --distribution-id E1L7QZR4LQV4LE \
  --id <InvalidationId> \
  --query 'Invalidation.Status' \
  --output text
```

## 배포 후 확인

```bash
# index.html이 CloudFront에서 200으로 내려오는지 확인
curl -fsSI https://hibot-docu.com | sed -n '1,20p'

# API 연결 대상도 같이 확인
curl -fsS https://api.hibot-docu.com/api/menus | head -c 500
```

브라우저에서 `https://hibot-docu.com` 접속 후 강력 새로고침으로 새 asset을 확인합니다.

## 자주 나는 문제

| 증상 | 확인할 것 |
|---|---|
| `InvalidAccessKeyId`, `InvalidClientTokenId` | `aws sts get-caller-identity` 실패 여부, 로컬 AWS credential 만료 여부 |
| 배포 후 화면이 예전 그대로 | CloudFront invalidation 완료 여부, 브라우저 캐시 |
| API가 로컬/잘못된 주소로 호출됨 | `.env.production` 값 확인 후 다시 빌드 |
| `vite build`는 성공했지만 경고가 있음 | chunk size 경고는 배포 차단 사유는 아님. 실제 오류만 중단 처리 |

## 주의

- `dist/`는 빌드 산출물이므로 수동 수정하지 않습니다.
- 기본 스크립트는 오래된 hashed asset을 삭제하지 않습니다. 예전 HTML을 들고 있는 브라우저가 구 asset을 요청할 수 있기 때문입니다.
- S3 정리가 필요할 때만 `PRUNE_STALE_ASSETS=true ./scripts/deploy-frontend.sh`를 사용합니다.
- AWS Key를 문서나 커밋에 남기지 않습니다.
