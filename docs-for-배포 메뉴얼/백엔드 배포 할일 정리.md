# 백엔드 배포 할일 정리

## 방향

- 먼저 `EC2`부터 생성하는 게 맞다.
- 현재 구조는 `NestJS + SQLite + S3 업로드`라서, 백엔드는 일단 EC2 1대로 시작 가능하다.
- 프론트는 나중에 `S3 + CloudFront`로 분리 배포하면 된다.

## 1. EC2 생성

- `t3.small` 또는 여유가 있으면 `t3.medium`으로 생성
- OS는 `Ubuntu 24.04 LTS` 또는 `Ubuntu 22.04 LTS`
- 보안그룹 오픈
- `22` SSH
- `80` HTTP
- `443` HTTPS
- 탄력 IP 연결 여부 결정

## 2. 서버 기본 세팅

- 접속용 키페어 준비
- 서버 접속 후 시스템 업데이트
- `node`, `pnpm`, `nginx`, `pm2`, `git` 설치
- 타임존 `Asia/Seoul` 확인

## 3. 프로젝트 배포 준비

- 레포 클론
- 백엔드 디렉터리 기준으로 설치 및 빌드
- `.env` 파일 작성
- 필수 환경변수 확인
- `PORT`
- `DATABASE_FILE`
- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`
- `AWS_S3_BUCKET_NAME`
- `AWS_S3_REGION`
- 민감정보는 깃에 올리지 않기

## 4. SQLite 운영 방식 결정

- 현재 DB는 `SQLite` 파일 기반
- EC2 내부 디스크에 DB 파일 둘 경로 고정
- 백업 정책 필요
- 최소 하루 1회 DB 파일 백업
- 배포 전 DB 파일 복사본 남기기
- 이후 트래픽이나 안정성 요구가 커지면 `RDS(Postgres)` 전환 검토

## 5. 백엔드 실행

- `pm2`로 Nest 서버 실행
- 서버 재부팅 시 자동 시작 설정
- `localhost:3000`에서 먼저 정상 동작 확인
- `/api` 라우트 응답 확인
- `/api/upload/presign`도 운영 환경변수 기준으로 확인

## 6. Nginx 연결

- 외부 `80/443` 요청을 내부 Node 서버로 프록시
- `/api`를 백엔드로 연결
- CORS 허용 도메인은 프론트 배포 주소 기준으로 다시 정리
- 업로드 기능이 있으니 프록시 body size 제한도 확인

## 7. 도메인/SSL

- 도메인을 EC2 IP로 연결
- `certbot`으로 HTTPS 적용
- HTTPS 적용 후 API_BASE_URL 확정

## 8. 배포 후 점검

- 로그인 정상 동작
- 카테고리/프로토타입 조회 정상 동작
- 이미지 업로드 정상 동작
- 서버 재시작 후에도 서비스 유지 확인
- DB 파일 백업 동작 확인

## 현재 기준 추천 순서

1. EC2 생성
2. 서버 기본 세팅
3. 백엔드 실행 확인
4. Nginx + 도메인 + HTTPS 연결
5. 프론트 S3 배포
6. 프론트 도메인에서 백엔드 연동 확인
